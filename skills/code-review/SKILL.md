# 代码复盘与审查技能

对当前分支的代码变更进行**严格、逐行**的审查，目标是在代码合入前发现所有潜在 Bug 和隐患。

## 触发条件

以下场景自动触发：
- 用户说"审查代码"、"review"、"复盘"、"帮我看看代码有没有问题"
- 用户说"检查一下改动"、"上线前看看"
- 用户提供了 PR 链接或分支名要求审查

## 核心原则

**宁可误报，不可漏报。** 审查的目的是拦截线上事故，不是为了好看。每一条发现都必须说清楚"在什么输入/场景下会出问题"。

## 执行流程

### 第一步：收集变更范围

按以下优先级确定审查范围：

1. 如果用户指定了文件或 PR，审查指定内容
2. 否则执行 `git diff HEAD`，如果为空则 `git diff HEAD~1`
3. 执行 `git log --oneline -5` 了解最近提交上下文
4. 执行 `git diff --stat` 获取变更文件清单和规模

### 第二步：深度阅读代码（最关键）

**禁止只看 diff 就下结论。** 必须：

1. **读完整文件**：对每个变更文件，用 Read 工具读取完整内容，理解变更在完整上下文中的位置
2. **读上下游依赖**：
   - 被修改方法的调用方（grep 方法名找到所有调用点）
   - 被修改类的父类/接口
   - 相关的配置文件（Spring XML、MyBatis Mapper、application.yml 等）
   - 数据库表结构定义（如有 DDL 或 migration 文件）
3. **读测试文件**：确认变更是否有对应的测试覆盖
4. **还原修改前逻辑**：通过 `git show HEAD~1:filepath` 对比理解改了什么

### 第三步：逐项审查（检查清单）

对每个变更文件，按以下维度逐一检查。**每个维度都必须过一遍**，不能跳过：

#### A. 逻辑正确性
- [ ] 条件判断是否覆盖了所有分支？是否存在遗漏的 else / default case？
- [ ] 循环的边界条件：空集合、单元素、超大集合是否都能正确处理？
- [ ] 递归是否有明确的终止条件？是否可能栈溢出？
- [ ] 返回值是否在所有路径上都有正确赋值？
- [ ] 业务逻辑是否与需求一致？（对照注释、文档、提交信息）

#### B. 空指针与空值
- [ ] 从 Map/数据库/外部接口获取的值，使用前是否做了 null 检查？
- [ ] Optional 的使用是否正确？是否存在 `.get()` 不带 `isPresent()` 检查？
- [ ] 集合操作前是否检查了 null 和 empty？
- [ ] 字符串比较是否用了 `Objects.equals()` 或常量在前？
- [ ] 链式调用中间是否可能产生 null？

#### C. 并发与线程安全
- [ ] 共享可变状态是否有正确的同步保护？
- [ ] 是否存在 check-then-act 竞态条件？
- [ ] 数据库操作是否在正确的事务边界内？
- [ ] 分布式锁的使用是否正确（获取、续期、释放）？
- [ ] 线程池的配置是否合理？是否可能耗尽？

#### D. 数据库与 SQL
- [ ] 新增 SQL 是否走索引？是否可能全表扫描？
- [ ] 批量操作的数据量是否有上限控制？
- [ ] 是否存在 N+1 查询问题？
- [ ] SQL 拼接是否使用参数化查询（防 SQL 注入）？
- [ ] 事务范围是否最小化？是否有长事务风险？
- [ ] UPDATE/DELETE 语句是否有 WHERE 条件？是否可能误操作全表？

#### E. 异常处理
- [ ] catch 块是否吞掉了异常而没有记录日志？
- [ ] 受检异常的处理是否合理（不是简单 catch + ignore）？
- [ ] 资源是否在 finally / try-with-resources 中正确关闭？
- [ ] 异常信息是否包含足够的上下文用于排查？
- [ ] 是否存在过于宽泛的 catch (Exception e)？

#### F. 安全性
- [ ] 用户输入是否经过验证和清洗？
- [ ] 是否存在越权访问的可能（水平越权/垂直越权）？
- [ ] 敏感信息（密码、Token、密钥）是否硬编码或打印到日志？
- [ ] 文件操作是否存在路径遍历风险？
- [ ] 接口是否缺少鉴权/限流？

#### G. 性能
- [ ] 是否在循环中做了数据库查询或 RPC 调用？
- [ ] 大对象是否及时释放？是否可能 OOM？
- [ ] 是否创建了不必要的对象（如循环内 new 大对象）？
- [ ] 缓存的使用是否合理？是否存在缓存穿透/雪崩风险？
- [ ] 日志打印是否在热路径上做了字符串拼接（应该用占位符）？

#### H. 兼容性与影响范围
- [ ] 接口签名的变更是否向后兼容？
- [ ] 数据库表结构变更是否向后兼容？是否需要分步发布？
- [ ] 配置变更是否有默认值兜底？
- [ ] 是否影响了其他模块的现有功能？（检查调用方）
- [ ] 枚举/常量的变更是否同步更新了所有引用点？

#### I. 代码规范
- [ ] 命名是否清晰准确？是否有误导性命名？
- [ ] 方法长度是否合理（超过 80 行需要拆分）？
- [ ] 是否有重复代码可以抽取？
- [ ] 魔法数字是否替换为常量？
- [ ] 注释是否与代码一致（是否有过时注释）？

### 第四步：输出审查报告

报告格式如下：

```markdown
# 代码审查报告

## 审查概要

| 项目 | 内容 |
|------|------|
| 审查范围 | xxx分支 / PR #xxx |
| 变更文件数 | N 个 |
| 变更行数 | +xxx / -xxx |
| 审查时间 | 2026-xx-xx |
| 风险等级 | 🔴 高危 / 🟡 中危 / 🟢 低危 |

## 变更文件清单

| 文件 | 变更类型 | 行数 | 风险 |
|------|---------|------|------|
| path/to/file | 修改 | +10/-5 | 🟡 |

## 🔴 必须修复（阻塞合入）

### [P0-001] 问题标题
- **文件**：`path/to/file:行号`
- **类型**：逻辑缺陷 / 空指针 / SQL注入 / ...
- **问题描述**：具体说明在什么场景下会出问题
- **复现条件**：当 xxx 输入为 yyy 时
- **影响**：会导致什么后果
- **修复建议**：
  ```java
  // 建议的修复代码
  ```

## 🟡 建议修复（不阻塞但应排期）

### [P1-001] 问题标题
（格式同上）

## 🟢 优化建议（可选）

### [P2-001] 问题标题
（格式同上）

## ✅ 审查通过项

简要列出检查过且没有问题的维度，证明审查的覆盖度。

## 审查结论

- [ ] 可以合入 ✅
- [ ] 需要修复后重新审查 🔄
- [ ] 存在严重问题，建议重新设计 ❌
```

### 第五步：保存报告

1. 将报告保存到项目根目录，命名为 `code-review-{日期}.md`
2. 如果有 🔴 必须修复的问题，**明确告知用户不建议合入**，逐条列出原因
3. 询问用户是否需要自动修复发现的问题

## 审查红线（发现即阻塞）

以下问题无论什么情况都必须标记为 🔴 P0：

1. **数据丢失风险**：DELETE/UPDATE 无 WHERE、DROP TABLE、truncate
2. **资金损失风险**：金额计算用 float/double 而非 BigDecimal
3. **安全漏洞**：SQL 注入、XSS、硬编码密钥、未鉴权接口
4. **线上故障风险**：死循环、无限递归、未捕获异常导致进程退出
5. **数据不一致**：缺少事务保护的多表写入、并发修改无锁
6. **资源泄漏**：未关闭的连接/流/锁

## 自审（Meta-Review）

输出报告前，对自己的审查做一次自检：

1. 是否每个变更文件都覆盖到了？
2. 是否读了完整文件而不只是 diff？
3. 是否检查了调用方和上下游影响？
4. 是否有主观偏见导致的漏检（比如觉得"这段代码看起来没问题"就跳过了）？
5. 发现的每个问题是否都给出了具体的复现条件和修复建议？

如果自检发现遗漏，回去补充后再输出报告。

## 关键约束

- **不允许输出"代码整体看起来没问题"这种敷衍结论** — 必须逐维度说明检查结果
- **不允许只看 diff 不看完整文件** — 上下文缺失是误判的最大来源
- **不允许跳过任何检查维度** — 每个维度至少输出一行结论
- **发现问题必须给修复代码** — 光说"这里有问题"没有价值，要给出能直接用的修复方案
- **中文输出，代码保持英文**
