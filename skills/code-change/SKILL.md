# 代码变更标准工作流

所有涉及代码修改的需求，必须遵循"先方案后编码"的两阶段流程。

## 触发条件

当用户提出需要修改代码的需求时自动触发，包括但不限于：功能开发、性能优化、重构、Bug修复等。

## 阶段一：技术方案

### 执行流程

1. **阅读当前代码**：读取需求涉及的所有相关文件，理解现有逻辑
2. **分析方案**：设计实现方案，评估影响范围
3. **输出技术方案文档**（Markdown 格式），内容包括：
   - **背景与问题**：当前代码的现状和痛点
   - **方案设计**：核心思路、改动点说明
   - **涉及文件清单**：表格列出每个文件的变更类型和说明
   - **代码示例**：关键代码片段（伪代码或实际代码）
   - **影响分析**：兼容性、风险点、容错降级
   - **性能/效果分析**：量化对比（如适用）
4. 文档保存到项目根目录，文件名与需求相关

### 输出要求

- 中文撰写，代码和技术术语保持英文
- 基于**当前实际代码**编写，不引用不存在的方法或结构
- 代码片段用语法高亮（```java / ```xml / ```sql）
- 对比性内容使用 Markdown 表格
- 量化分析用具体数字

### 上传到 JoySpace

技术方案 Markdown 文件保存到项目根目录后，自动执行以下命令将文档同步上传到 JoySpace：

```bash
node ~/.claude/skills/markdown-to-joyspace/scripts/import_markdown_doc.js --file <md文件绝对路径>
```

上传成功后，将返回的 JoySpace 链接告知用户。如果上传失败，提示失败原因但不阻塞后续流程。

### 等待确认

方案输出后，明确告知用户"方案已输出并上传到 JoySpace，请确认后我再生成代码"，附上 JoySpace 文档链接，**不主动开始编码**。

## 阶段二：代码实现

用户确认方案后，再执行代码编写：

1. 按方案逐文件实现改动
2. 同步更新技术方案文档（如实现中有调整）
3. 尝试编译验证（如项目支持本地编译）

## 关键约束

- **绝不跳过方案阶段直接写代码**
- **方案必须基于当前代码实际状态**，先 Read 文件再写方案
- 用户说"确认"/"可以"/"开始写"等才进入阶段二
