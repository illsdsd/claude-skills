# 技术文档生成技能

根据当前分支的本地代码修改，自动生成一份结构完整、内容详尽的技术优化文档（Markdown 格式）。

## 触发条件

当用户要求根据本次修改/变更生成技术文档、优化文档、设计文档时触发。

## 执行流程

### 第一步：收集变更信息

1. 执行 `git diff HEAD` 查看所有已暂存和未暂存的修改
2. 如果 diff 为空，尝试 `git diff HEAD~1` 或 `git log --oneline -5` 确认最近提交
3. 识别所有变更文件的完整路径和变更类型（新增/修改/删除）

### 第二步：深度阅读代码

1. **逐个读取所有变更文件的完整内容**（不仅仅是 diff），理解完整上下文
2. 读取变更文件的相关依赖文件（如父类、接口、调用方），理解修改在整体架构中的位置
3. 对于 MyBatis XML 等配置文件，理解 SQL 语句的具体作用和索引利用情况
4. 理解优化前的代码逻辑（从 diff 的删除部分和基类默认实现中还原）

### 第三步：生成文档

文档必须包含以下章节结构：

#### 1. 概述
- **背景**：描述业务场景、数据规模、原有实现的痛点
- **优化目标**：明确列出本次优化要解决的问题
- **策略总览**：用表格概括所有优化策略

#### 2. 涉及模块与文件
- 表格形式列出所有变更文件，标注模块、文件路径、变更类型、变更说明

#### 3. 优化策略详解（每个策略独立小节）
每个策略必须包含：
- **优化前的问题**：附带原有代码片段，分析具体的性能瓶颈
- **性能瓶颈量化分析**：用表格列出各瓶颈点的影响和量级
- **优化后的实现**：附带新代码片段，标注关键设计点
- **设计要点**：解释关键技术决策的原因
- **性能收益量化**：用表格对比优化前后的指标

#### 4. 架构重构详解
- 类层次结构变化（ASCII 树形图展示）
- 模板方法/策略模式的应用
- 向后兼容性保证
- 依赖关系变更说明

#### 5. 核心数据流对比
- 用 ASCII 流程图分别展示优化前和优化后的完整数据流

#### 6. 可配置化参数
- 表格形式列出所有可配置参数：参数名、配置来源、默认值、说明、是否运行时可调
- 提供运行时调参的具体示例

#### 7. 容错与降级机制
- 每种容错机制的触发条件和处理链路
- 用 ASCII 图展示降级链路
- 线程安全保证说明

#### 8. 预期性能提升总结
- 总表对比所有优化项的优化前、优化后、预期提升倍数
- 综合预估整体提升效果

## 文档质量要求

1. **代码片段**：关键的优化前后代码必须附带，用 ```java 或 ```sql 等语法高亮
2. **量化分析**：尽量用具体数字（如"100倍"、"N次→1次"）而非模糊描述
3. **ASCII 图表**：架构图、流程图、降级链路图等使用 ASCII 字符绘制
4. **表格化呈现**：对比性内容优先使用 Markdown 表格
5. **中文撰写**：所有描述性文本使用中文，代码和技术术语保持英文
6. **不遗漏细节**：每个修改文件的每处变更都要在文档中有所体现

## 输出位置

将文档生成到项目根目录下，文件名根据修改内容命名，如 `XXX性能优化技术文档.md`。
